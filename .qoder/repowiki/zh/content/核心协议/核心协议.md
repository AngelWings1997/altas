## 项目结构
ALTAS Workflow 采用“主协议 + 分层技能 + 按需模块”的组织方式：
- 主协议位于顶层 SKILL.md，定义整体流程、触发词、规模评估、铁律与输出规范
- 参考资料索引 reference-index.md 提供统一发现入口，指导按需加载
- 子协议（如 RIPER-5、RIPER-DOC、SDD-RIPER-DUAL-COOP）提供严格模式、文档专家与双模型协作的范式
- 分层技能（Checkpoint-Driven、Superpowers）分别覆盖轻量检查点与测试驱动、系统化调试、子代理驱动等能力

## 核心组件
- 触发词与模式选择：通过关键词自动判定工作流深度与模式（FAST/DEEP/DEBUG/MULTI/DOC/MAP/ARCHIVE）
- 规模评估与自动升降级：根据任务复杂度与文件/行数阈值选择 XS/S/M/L，并支持执行中动态调整
- 铁律约束：No Spec No Code、No Approval No Execute、Evidence First、Root Cause Required、TDD 铁律等
- 检查点输出：XS/S 输出简要总结；M/L 输出完整检查点模板，含“已完成/当前/下一步/后续”与操作选项
- 渐进式披露：按阶段与规模加载参考文件，避免上下文污染
- 上下文装配策略：Hot/Warm/Cold 三层上下文，冲突/不确定时强制重读完整 Spec

## 架构总览
ALTAS Workflow 将 Spec-Driven、Checkpoint-Driven 与 Superpowers 融合，形成“输入准备 → 研究对齐 → 方案对比（L）→ 详细规划 → 执行实现（TDD/Subagent）→ 三轴审查 → 知识沉淀”的闭环。

```mermaid

B --> C["输入准备(PRE-RESEARCH)"]
C --> D["研究对齐(RESEARCH)"]
D --> E{"是否为 L 规模?"}
E --> |是| F["方案对比(INNOVATE)"]
E --> |否| G["详细规划(PAUSE-AND-WAIT)"]
F --> G
G --> H["执行实现(EXECUTE)<br/>TDD/Subagent"]
H --> I["三轴审查(REVIEW)"]
I --> J{"是否需要归档?"}
J --> |是| K["知识沉淀(ARCHIVE)"]
J --> |否| L["结束/待续"]
```

## 详细组件分析

### 4 级任务深度评估机制（XS/S/M/L）
- XS（极小）：typo/配置值，<10 行，跳过 micro-spec，事后一行 summary
- S（小）：1-2 文件，逻辑清晰，micro-spec（1-3 句），批准后执行
- M（中）：3-10 文件，模块内，轻量 Spec 落盘，Research→Plan→Execute（TDD）→Review
- L（大）：跨模块，>500 行，架构级，完整 Spec+Innovate+Archive，Research→Innovate→Plan→Execute（TDD/Subagent）→Review→Archive

自动升降级：执行中若复杂度超出预期，立即暂停并建议升级；用户可随时选择“升级为M/降级为S”。

```mermaid
flowchart TD
Start(["接收任务"]) --> Analyze["分析任务规模与复杂度"]
Analyze --> XS{"XS 规模?"}
XS --> |是| XSExec["直接执行→验证→1行summary"]
XS --> |否| SML{"S/M/L 规模?"}
SML --> |S| SPath["micro-spec→批准→执行→回写"]
SML --> |M| MPath["Research→Plan→Execute(TDD)→Review"]
SML --> |L| LPath["Research→Innovate→Plan→Execute(TDD/Subagent)→Review→Archive"]
SPath --> Monitor["执行监控与自动升降级"]
MPath --> Monitor
LPath --> Monitor
Monitor --> Upgrade{"复杂度上升?"}
Upgrade --> |是| Pause["暂停并建议升级/降级"]
Upgrade --> |否| Continue["继续执行"]
Pause --> Continue
Continue --> End(["完成/等待下一步"])
```

### 工作流状态机控制逻辑
- 标准模式：PRE-RESEARCH → RESEARCH → INNOVATE（L）→ PLAN → EXECUTE → REVIEW → ARCHIVE
- FAST 模式：跳过 Research/Plan，直接执行，事后同步 Spec
- DEBUG 模式：无根因不修复，诊断/验证子模式
- MULTI 模式：多项目协作，作用域 local/CROSS
- DOC 模式：Absorb→Outline→Author→Fact-Check
- MAP 模式：只读 CodeMap，必要时升级为标准流程
- ARCHIVE 模式：双视角归档（human/llm）

### 铁律约束执行机制
- No Spec, No Code（Size XS 豁免）
- No Approval, No Execute
- Spec is Truth（Spec 与代码冲突时，代码是错的）
- Reverse Sync（偏差暴露→先更新 Spec→再修代码）
- Evidence First（完成由验证结果证明）
- No Fixes Without Root Cause（根因分析前置）
- TDD Iron Law（Size M/L：无失败测试不写生产代码）
- Resume Ready（长任务暂停前留恢复锚点）

### 检查点标准化输出格式
- XS：1 行 summary（完成什么 + 验证结果）
- S：短检查点（当前理解/核心目标/下一步）
- M/L：完整检查点模板（已完成/当前/下一步/后续、当前成果、预期产出、下一步操作）

## 性能考量
- 通过 Hot/Warm/Cold 上下文策略降低无效 token 消耗
- Checkpoint-Driven 轻量锚点减少常驻上下文，按需模块加载
- TDD/Subagent 驱动提升质量与迭代效率，减少后期调试成本
- 按需加载避免一次性加载大量参考文件导致上下文污染

## 故障排查指南
- 铁律违规：No Spec No Code、No Approval No Execute、Evidence First、Root Cause Required、TDD 铁律
- 执行偏差：Reverse Sync（先更新 Spec→再修代码→重对齐）
- 上下文异常：冲突/不确定时强制重读完整 Spec
- 调试流程：系统化调试四阶段（根因调查→模式分析→假设与测试→实现），无根因不修复

## 结论
ALTAS Workflow 通过 4 级规模评估与渐进式披露机制，将 Spec-Driven、Checkpoint-Driven 与 Superpowers 有机融合，形成可扩展、可验证、可沉淀的工程化工作流。开发者应遵循铁律约束，按需加载参考文件，严格执行检查点输出与三轴审查，确保交付质量与可维护性。
