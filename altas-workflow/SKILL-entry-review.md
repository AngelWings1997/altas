## 一、当前版本仍可改进的问题

### A. 原子化拆解要求分散，缺少显式交叉引用

- **优先级：高**
- **现状：** 原子化拆解要求分布在三个独立章节（铁律#1 第56行、首轮模板第227-237行、拆解要求第243-253行、PLAN门禁第316-323行），彼此之间没有显式链接。
- **风险：** 首次阅读的 Agent 可能只看到其中一处，未意识到三处互相强化；尤其是首轮模板中的"当前原子步骤清单"看似可选，实际是必填。
- **建议：**
  - 在"已有明确任务"小节（第206行之后）加一条显式指引，指向"从接收任务到 PLAN 的拆解要求"章节
  - 在首轮模板中将"当前原子步骤清单"标题改为"当前原子步骤清单 **[必填]**"，并注明"即使 XS/S 也不豁免此结构"

### B. 首轮模板未明确 XS/S 规模的原子拆解义务

- **优先级：高**
- **现状：** 规模评估表（第135-136行）中 XS 写"跳过，事后 1 行 summary"，S 写"micro-spec（1-3 句）"，但首轮模板的原子步骤清单没有按规模区分要求。
- **风险：** Agent 可能误以为 XS/S 不需要输出原子步骤清单，与"从接收任务到 PLAN 的拆解要求"中的"必须持续输出原子化拆解"产生矛盾。
- **建议：** 在首轮模板的"当前原子步骤清单"处添加说明：XS 可精简为 1-2 个步骤但结构不变；S 至少覆盖核心步骤。

### C. 不确定性处理缺少视觉强调

- **优先级：中**
- **现状：** 铁律#10 和问题升级机制（第350-365行）内容完整且无歧义，但在长文档中容易被快速扫描跳过。
- **风险：** 受时间压力驱动的 Agent 可能在快读时忽略铁律#10 的强制性。
- **建议：** 在"异常与恢复"章节的问题升级机制后，增加一个醒目的 callout 块，明确列出"不允许的行为"（猜测意图继续、选最可能方案继续、假设依赖存在继续）和"必须的行为"（输出状态、描述问题、提出选项、暂停等待）。

### D. 路由冲突裁决缺少决策树

- **优先级：中**
- **现状：** 路由冲突优先级（第120-129行）用文字描述了四级优先级，但没有可视化决策树。
- **风险：** 多触发词同时命中时，Agent 需要反复对照文字段落来裁决，增加误判概率。
- **建议：** 在第129行后增加一个简单的文本决策树：显式触发词 → 只读门禁 → 特殊模式 → 默认 Coding，每个节点 YES/NO 分支清晰。

### E. PLAN 阶段门禁未突出 writing-plans 技能的关联

- **优先级：低**
- **现状：** PLAN 阶段门禁（第324行）提到读取 `references/superpowers/writing-plans/SKILL.md`，但仅作为末尾一行参考，没有说明该技能本身对原子任务结构有额外要求。
- **风险：** Agent 可能跳过该技能的加载，导致 Plan 质量不如预期。
- **建议：** 将第324行改为更醒目的提示，说明 writing-plans SKILL 是 PLAN 阶段的**必读**参考，且其中包含计划质量标准。

### AF. 缺少入口文案一致性的自动校验

- **优先级：低**
- **现状：** 仓库目前只有 `scripts/validate_aliases_sync.py` 用于校验 `aliases.md` 与 `SKILL.md` frontmatter `trigger_keywords` 一致；像版本号、初始化提示文案、首轮响应字段这类入口关键字面量仍靠人工维护。
- **风险：** 本轮就出现过 `SKILL.md` 版本已到 `4.4`，初始化提示仍写 `v4.1` 的漂移，后续仍可能重复发生。
- **建议：** 扩展校验脚本，至少检查 `frontmatter version`、初始化提示版本、必备首轮字段、关键章节标题、所有引用文件路径是否存在。
