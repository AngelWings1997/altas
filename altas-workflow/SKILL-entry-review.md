# SKILL Entry Review

## 背景

本文件记录对 `SKILL.md` 作为技能入口的评审结论，以及本次直接实施的重构方向。

## 主要问题

### 1. 入口过载

`SKILL.md` 之前同时承担了入口、完整协议、特殊模式手册、参考索引、示例集、产物治理等多种职责。

问题在于：
- 入口文件过长，不利于模型快速判断“现在是否该使用这个技能”
- 常驻 token 成本偏高，与仓库强调的“主体精简、细节按需加载”不一致
- 真正影响行为的入口约束被大量说明性内容稀释

### 2. 触发词和路由词典漂移

仓库中的 `README.md`、`QUICKSTART.md`、`workflow-quickref.md` 已经支持一些入口别名，例如：
- `PROJECT MAP`
- `MAP ALL`
- `REVIEW SPEC`
- `REVIEW EXECUTE`
- `CROSS`
- `EXIT ALTAS`

但旧版 `SKILL.md` 的 frontmatter 和入口速查没有完整覆盖这些别名，容易导致用户文档和技能入口不一致。

### 3. Description 不够 discovery-first

旧版 frontmatter `description` 混合了“做什么”和“怎么做”的流程摘要。

这类描述虽然信息多，但对技能发现不够友好，也可能让模型只看描述，不继续读技能正文。

### 4. 信息重复导致维护成本高

旧版 `SKILL.md` 里重复了大量本应由这些文件承担的内容：
- `reference-index.md`
- `README.md`
- `QUICKSTART.md`
- 各 `references/...` 子文档

重复信息越多，越容易出现后续漂移。

### 5. 规则优先级写法偏激进

旧版用了“本 Skill 优先于通用行为准则”的表达。对于跨平台技能入口，这种写法容易和宿主平台的系统规则、项目规则产生优先级歧义。

更稳妥的表达方式应该是：
- 本 Skill 补充工程工作流约束
- 若与宿主平台系统安全规则冲突，优先遵守更严格的规则
- 若与项目规则冲突，优先采用更保守、更安全的执行方式

## 本次已实施的调整

### 1. 把 `SKILL.md` 重构为真正的 Bootstrap 入口

保留入口最需要的内容：
- 何时使用
- 不何时使用
- 核心铁律
- 路由和规模判断
- 首轮响应契约
- 检查点输出契约
- 按需加载索引

下沉长篇细节，避免入口继续承担“完整协议手册”的职责。

### 2. 统一触发词和别名

在 `SKILL.md` frontmatter 与入口速查中统一纳入：
- `PROJECT MAP`
- `MAP ALL`
- `REVIEW SPEC`
- `REVIEW EXECUTE`
- `CROSS`
- `EXIT ALTAS`

这样入口与现有仓库文档的词典保持一致。

### 3. 重写 frontmatter `description`

新版描述改为以“何时使用”为中心，不再概述完整流程，提升技能发现性。

### 4. 收敛入口与参考资料的职责边界

新版 `SKILL.md` 明确说明：
- 本文件负责路由、门禁、首轮行为
- 细节模板与阶段策略去 `reference-index.md` 和 `references/` 按需读取

### 5. 调整规则优先级表述

新版不再强调“覆盖”通用规则，而是强调“补充工作流约束，并优先遵守更严格的安全规则”。

## 重构后的入口设计原则

1. 先让模型知道“这是什么、什么时候该读”
2. 再让模型知道“读了以后第一步该怎么做”
3. 最后只给必要的跳转，不把全部细节塞进入口

## 后续仍可继续优化的点

- 可以把“产物命名约定”再拆到单独参考文件，进一步压缩入口体积
- 可以新增一份 `references/entry/aliases.md`，集中维护触发词别名，降低重复修改成本
- 可以补一份“入口瘦身前后对照”文档，方便团队理解为何要这样改
