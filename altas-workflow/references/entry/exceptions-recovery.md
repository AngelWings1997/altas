# 异常与恢复

> 本文件从 `SKILL.md` 移出，以保持主入口精简。
> Agent 在遇到异常、不确定、或需要退出协议时加载本文件。

## 问题升级机制

当遇到以下情况时，**必须暂停并找用户确认**（遵守铁律#10），禁止擅自决策或跳过：

| 情况 | 示例 | 处理方式 |
|------|------|----------|
| 需求不明确 | 无法理解用户意图 | 输出澄清问题，等待确认 |
| 技术方案不确定 | 多个可行方案无法取舍 | 列出方案对比，请求决策 |
| 遇到阻塞 | 工具失败、权限不足、依赖缺失 | 输出问题详情，请求协助 |
| 发现隐藏复杂度 | 实际复杂度超出规模评估 | 提议升级规模，重新评估 |
| 与 Spec 冲突 | 实现中发现 Spec 不合理 | 暂停执行，请求 Spec 修正 |

**升级流程：**
1. 立即输出检查点（当前状态、已完成、阻塞原因）
2. 清晰描述问题：发生了什么、为什么阻塞、需要什么帮助
3. 提出选项：列出可行的下一步方案及建议
4. 暂停等待用户确认

## 不确定时的强制行为（铁律 #10 执行细则）

**不允许的行为：**
- ❌ 猜测用户意图然后继续
- ❌ 选择"最有可能"的方案然后继续
- ❌ 假设某个依赖存在然后继续
- ❌ 延后确认风险或边界条件

**必须的行为：**
- ✅ 立即输出当前状态（已完成、当前阻塞、为什么卡住）
- ✅ 清晰描述问题：发生了什么、为什么无法继续、需要什么帮助
- ✅ 提出可选下一步方案（如有多个）
- ✅ 暂停等待用户确认，不自主决策

## EXIT ALTAS 规范

### 主动退出（用户输入 `EXIT ALTAS`）

退出前必须输出完整恢复锚点：

| 字段 | 内容 |
|------|------|
| **当前阶段** | PRE-RESEARCH / RESEARCH / INNOVATE / PLAN / EXECUTE / REVIEW / ARCHIVE |
| **已完成** | 本轮产出的文件、结论、下一步待办 |
| **待办** | 未完成项及优先级 |
| **恢复锚点** | 基于 Spec + 代码 + 对话历史的最小重建路径 |

### 强制中断（非用户主动：上下文耗尽/工具失败等）

退出前输出最小恢复锚点：

| 字段 | 内容 |
|------|------|
| **中断类型** | `[FORCED] 非用户主动退出` |
| **当前阶段** | 同上 |
| **最后检查点** | 最近一次 checkpoint 的阶段和产出摘要 |
| **Spec 状态** | `完整` / `部分` / `[RECOVERED]` 重建 |
| **最小恢复路径** | 仅含 Spec 路径 + 最后 Checkpoint + 未完成 Checklist 项 |

**Spec 损坏时**（丢失或不一致）：基于代码现状和对话历史重建最小 Spec，标记 `[RECOVERED]` 后请求确认。

## 能力降级

| 缺失能力 | 降级方案 |
|----------|----------|
| Subagent / 并行 Agent | 单会话 + 原子 Checklist + 常规检查点 |
| Todo 面板 | 文本化 Checklist + 定期 checkpoint 输出 |
| `create_codemap` / `build_context_bundle` | 手动读取文件并生成上下文摘要 |
| `sdd_bootstrap` | 手动执行 Research → 直接写 Spec |
| 上下文窗口不足 | 立即执行 Resume Ready，状态写回 Spec 后再继续 |
| Markdown 输出受限 | 降级为纯文本结构化输出 |

- 工具缺失不应阻塞主流程，但必须明确说明降级行为
- 降级后的替代方案应在本轮 checkpoint 中标注

## 其他异常

- TDD 红灯连续失败：暂停并给出根因候选，不继续盲改。
- Reverse Sync 频繁出现：提议重做 Plan 或升级规模。
- 上下文将满：立即执行 Resume Ready，将状态写回 Spec 后再继续。

## 上下文窗口管理 (Context Window Management)

> **触发条件**: L 规模任务、长对话、或加载大量参考文件时必须关注上下文使用效率。

### 核心原则

| 原则 | 说明 |
|------|------|
| **最小上下文** | 只加载当前阶段必需的参考文件，不预加载未来阶段的内容 |
| **早释早放** | 已完成的阶段参考文件应在检查点中标注 `[可释放]`，避免占用上下文 |
| **分层加载** | 按规模分级决定加载量：XS/S 最小、M 标准、L 完整但需监控 |
| **Resume Ready** | 上下文 >80% 时必须输出恢复锚点，准备分段执行 |

### 分层加载策略

| 规模 | 加载策略 | 典型文件数 | 监控频率 |
|------|----------|-----------|----------|
| **XS** | 仅核心铁律 + 首轮响应模板 | 1-2 个 | 无需监控 |
| **S** | 核心铁律 + 对应模式协议 + spec-lite-template | 3-5 个 | 每个检查点 |
| **M** | 标准集（见 reference-index.md Size M 索引）| 6-10 个 | 每完成一个 Checklist 项 |
| **L** | 完整集（见 reference-index.md Size L 索引）| 15+ 个 | **持续监控** |

### 上下文健康检查清单

在每个检查点时，自检以下问题：

- [ ] 当前已加载的参考文件是否都是当前阶段必需？
- [ ] 是否有已完成阶段的文件可以释放？（标注 `[可释放]`）
- [ ] 下一阶段的计划是否已落盘到 Spec/Plan 文件中？（而非仅存在于对话上下文）
- [ ] TodoWrite 中的任务状态是否与实际进度一致？

### Resume Ready 触发条件

当满足以下任一条件时，**必须**输出恢复锚点：

| 信号 | 触发阈值 | 动作 |
|------|----------|------|
| 对话轮次过多 | 单任务对话 > 30 轮 | 输出检查点 + 恢复锚点 |
| 检查点累积 | 本任务已输出 > 5 个检查点 | 同上 |
| 文件修改量大 | 已修改/创建 > 10 个文件 | 同上 |
| 上下文感知 | 平台提示上下文即将耗尽 | **立即**输出恢复锚点 |

### 恢复锚点模板

```markdown
### 📍 Resume Ready — 恢复锚点

**任务**: <一句话描述>
**当前阶段**: <PRE-RESEARCH / RESEARCH / ...>
**规模**: <XS/S/M/L>

#### 已完成
- [x] <完成的步骤 1> → 产出: <文件/结论>
- [x] <完成的步骤 2> → 产出: <文件/结论>

#### 进行中
- [→] <当前步骤> → 进度: <X%> → 下一步: <具体动作>

#### 待办
- [ ] <待办项 1>
- [ ] <待办项 2>

#### 关键决策记录
1. <决策 1>: <选择了什么方案，为什么>
2. <决策 2>: ...

#### 可释放的参考文件
- `references/xxx.md` (已在 Phase X 使用完毕)

#### 下次恢复时的最小加载集
1. SKILL.md (路由 + 规模)
2. <当前阶段必需的参考文件>
3. Spec/Plan 文件 (最新版本)
```

### Red Flag: 上下文滥用

| Red Flag | → **STOP** | 风险 |
|----------|------------|------|
| 一次性加载 >10 个参考文件？ | 只加载当前阶段需要的 | 上下文溢出 |
| 反复重新读取同一文件？ | 将关键信息提取到 Spec 或笔记 | 浪费上下文 |
| 在对话中维护大型数据结构？ | 落盘到文件，仅保留引用 | 内存压力 |
| 检查点越来越简略？ | 强制使用完整模板，包含所有字段 | 信息丢失风险 |
