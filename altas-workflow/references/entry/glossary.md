# 术语表 (Glossary)

ALTAS Workflow 核心术语与概念定义，供新 AI 和用户快速理解。

---

## 核心概念

| 术语 | 定义 |
|------|------|
| **Spec** | 规格文档，是持久化上下文与压缩记忆层。包含功能定义、边界条件、验收标准、架构设计、测试策略等。所有 M 级以上任务必须先写 Spec 再写代码。 |
| **Spec Lite** | 轻量级 Spec，用于 S 级小任务。比标准 Spec 简化结构，但仍须包含 Acceptance Criteria。 |
| **CodeMap** | 项目代码地图，用于梳理入口文件、核心服务、数据流和相关测试的结构化文档。 |
| **Plan** | 任务拆解文档，将 Spec 拆分为可执行的步骤序列。 |
| **Archive** | 归档文档，记录已完成任务的关键决策、根因分析、解决方案和经验教训。 |

## 协议与模式

| 术语 | 定义 |
|------|------|
| **SDD** | Spec-Driven Development，规格驱动开发。先写 Spec，后写代码的核心方法论。 |
| **RIPER** | 五阶段协议：Research（研究）→ Innovate（创新）→ Plan（规划）→ Execute（执行）→ Review（审查）。 |
| **SDD-RIPER-ONE** | 完整工作流协议，融合 SDD 与 RIPER，包含 Pre-Research 前置阶段。 |
| **RIPER-ONE-Light** | 轻量版 RIPER，基于检查点模式运行，不需要完整的 Pre-Research。 |
| **RIPER-5** | 纯 RIPER 协议（无 SDD），适用于高风险代码修改。 |
| **RIPER-DOC** | 文档撰写模式下的 RIPER 协议。 |
| **SDD-RIPER-DUAL-COOP** | 双模型协作协议，使用两个 LLM 分别承担不同角色。 |

## 触发词

| 触发词 | 定义 |
|--------|------|
| **`>>`** | 极速通道，用于 XS 级单点修改（常量、文案、配置等）。 |
| **`FAST`** | 快速模式，用于 S 级小功能/小 bug/小范围增强。 |
| **`sdd_bootstrap`** | 标准启动，用于 M 级新增接口/模块/跨文件功能，触发完整 SDD-RIPER-ONE 流程。 |
| **`DEEP`** | 深度模式，用于 L 级架构重构/大范围迁移/跨模块改造。 |
| **`MAP`** / **`PROJECT MAP`** | 项目地图，用于快速了解项目结构、入口和数据流。 |
| **`DEBUG`** | 调试模式，用于线上报错/日志分析/性能异常。 |
| **`TEST`** | 测试模式，用于测试策略制定/测试计划生成/测试代码编写。 |
| **`PRD`** | 需求分析模式，用于将模糊需求转化为结构化 PRD。 |
| **`PERF`** | 性能模式，用于性能测试和分析。 |
| **`REVIEW`** | 审查模式，用于代码/实现三轴评审。 |
| **`REVIEW SPEC`** | Spec 评审，用于执行前审查 Spec/Plan 文档质量。 |
| **`REVIEW EXECUTE`** | 执行后评审，用于对已完成实现进行三轴评审。 |
| **`REF`** / **`REFACTOR`** | 重构模式，用于代码重构。 |
| **`MIGRATE`** | 迁移模式，用于框架/库/版本升级迁移。 |
| **`DOC`** | 文档模式，用于文档撰写。 |

## 规模等级

| 等级 | 定义 | 典型特征 |
|------|------|----------|
| **XS** | 单点修改 | 改一个常量/文案/配置，不涉及逻辑变更 |
| **S** | 小功能/小修复 | 改 1-3 个文件，逻辑明确，无需完整 Spec |
| **M** | 新功能/模块 | 跨多个文件，需要 Spec + Plan + 测试 |
| **L** | 架构级/大范围 | 涉及架构变更/大规模重构/迁移 |
| **复杂 M** | 逻辑复杂的 M 级 | 文件数不多但业务逻辑复杂，需要完整流程 |

## 模式内控制词

| 控制词 | 定义 |
|--------|------|
| **MULTI** | 多项目协作控制词，用于跨仓库/跨项目任务。 |
| **CROSS** | MULTI 的别名。 |
| **SWITCH** | 切换子项目上下文。 |
| **SCOPE LOCAL** | 局部作用域，限制当前操作仅影响指定范围。 |
| **REGISTRY** | 注册新子项目到多项目协作上下文。 |

## 工作流阶段

| 阶段 | 定义 |
|------|------|
| **PRE-RESEARCH** | 输入准备阶段，读取参考资料、CodeMap、上下文绑定。 |
| **RESEARCH** | 研究对齐阶段，写 Spec、定义边界和验收标准。 |
| **INNOVATE** | 方案对比阶段，设计技术方案并评估取舍。 |
| **PLAN** | 详细规划阶段，将方案拆解为可执行步骤。 |
| **EXECUTE** | 执行实现阶段，按 Plan 编写代码和测试。 |
| **REVIEW** | 审查阶段，三轴评审（Spec/实现/架构）。 |
| **ARCHIVE** | 知识沉淀阶段，记录关键决策和经验教训。 |

## 工具与系统

| 术语 | 定义 |
|------|------|
| **Superpowers** | 独立技能包集合，涵盖 TDD、Subagent、Debug、Brainstorming 等。 |
| **Subagent** | 子智能体，由主智能体派遣执行独立任务的 AI 实例。 |
| **TDD** | Test-Driven Development，测试驱动开发。先写测试，后写实现。 |
| **三轴评审** | 三个评审维度：Spec 评审（规格质量）、实现评审（代码质量）、架构评审（设计质量）。 |
| **Self-Improvement** | 自我进化机制，自动记录学习/错误/功能请求并晋升为工作流规则。 |

## 其他

| 术语 | 定义 |
|------|------|
| **Checkpoint-Driven** | 检查点驱动开发，将工作流分为多个检查点，每个检查点需要用户确认后才能继续。 |
| **Batch Override** | 批量覆盖，允许在批处理模式下跳过某些检查点以加速执行。 |
| **No Spec, No Code** | 铁律之一：没有 Spec 不允许写代码。 |
| **No Approval, No Execute** | 铁律之一：没有用户批准不允许执行。 |
