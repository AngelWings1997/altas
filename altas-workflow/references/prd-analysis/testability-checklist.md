# PRD 可测试性评审清单

> **调用时机**: PRD 分析阶段，测试工程师介入评审需求可测试性时加载
> **配合使用**: `references/prd-analysis/SKILL.md`（PRD 分析主流程）、`references/special-modes/test.md`（TEST 模式）

---

## 核心原则

**测试左移 (Shift-Left Testing)**: 在需求阶段发现可测试性问题，成本比在编码后修复低 10-100 倍。

### 可测试性定义

一个需求是可测试的，当且仅当：
1. **可观察**: 能够验证需求是否被满足
2. **可控制**: 能够设置测试所需的输入和状态
3. **可隔离**: 能够独立于其他需求进行测试
4. **可自动化**: 能够编写自动化测试脚本

---

## 评审时机

| 阶段 | 评审内容 | 参与角色 |
|------|----------|----------|
| **PRD Draft** | 需求完整性、边界条件 | 测试工程师 + 产品经理 |
| **PRD Review** | 验收标准、测试数据需求 | 测试工程师 + 开发 + 产品 |
| **设计评审** | 架构可测试性、测试钩子 | 测试工程师 + 架构师 |
| **开发启动前** | 测试计划、环境需求 | 测试工程师 + 开发 |

## 评审检查清单

### 1. 功能需求可测试性

#### 1.1 验收标准 (Acceptance Criteria)

- [ ] 每个功能需求都有明确的验收标准
- [ ] 验收标准使用 Given-When-Then 或类似格式
- [ ] 验收标准包含明确的输入和预期输出
- [ ] 验收标准可量化（非"快速"、"友好"等模糊描述）

**示例对比**:

```markdown
❌ 差的验收标准:
- 用户可以快速登录系统
- 系统应该友好地处理错误

✅ 好的验收标准:
- Given 用户输入正确的用户名和密码
- When 用户点击登录按钮
- Then 用户在 2 秒内被重定向到首页
- And 生成有效的 JWT token

- Given 用户输入错误的密码
- When 用户点击登录按钮
- Then 显示"用户名或密码错误"
- And 登录失败次数 +1
- And 同一 IP 5 分钟内失败 5 次后锁定 30 分钟
```

#### 1.2 边界条件定义

- [ ] 数值范围的边界（最小值、最大值、刚好超出边界）
- [ ] 字符串长度的边界（空字符串、最大长度、超长）
- [ ] 集合大小的边界（空集合、单元素、最大容量）
- [ ] 时间边界（开始时间、结束时间、时区切换）

**示例**:
```markdown
| 场景 | 输入 | 预期结果 |
|------|------|----------|
| 最小金额 | 0.01 元 | 允许创建订单 |
| 零金额 | 0 元 | 拒绝，返回错误码 E001 |
| 负数金额 | -10 元 | 拒绝，返回错误码 E001 |
| 最大金额 | 999,999,999.99 元 | 允许创建订单 |
| 超出最大 | 1,000,000,000 元 | 拒绝，返回错误码 E002 |
```

#### 1.3 异常场景覆盖

- [ ] 无效输入的处理方式
- [ ] 依赖服务不可用的降级策略
- [ ] 并发冲突的处理机制
- [ ] 资源耗尽（内存、磁盘、连接池）的应对

---

### 2. 非功能需求可测试性

#### 2.1 性能需求

- [ ] 响应时间指标（P50/P95/P99）
- [ ] 吞吐量指标（QPS/TPS）
- [ ] 并发用户数
- [ ] 测试数据规模（记录数、文件大小）
- [ ] 测试环境规格（与生产的比例）

**示例**:
```markdown
性能需求:
- P95 响应时间 < 200ms（1000 并发用户）
- 支持 5000 QPS 持续 1 小时
- 测试数据: 100万用户，1000万订单
- 测试环境: 生产环境的 1/4 规格
```

#### 2.2 安全需求

- [ ] 认证方式（OAuth2/JWT/API Key）
- [ ] 权限粒度（RBAC/ABAC）
- [ ] 敏感数据加密要求
- [ ] 安全扫描要求（OWASP Top 10）
- [ ] 审计日志要求

#### 2.3 兼容性需求

- [ ] 浏览器版本范围
- [ ] 移动端系统版本（iOS/Android）
- [ ] API 版本兼容策略
- [ ] 数据库版本兼容
- [ ] 第三方依赖版本

---

### 3. 测试数据需求

#### 3.1 数据类型

- [ ] 需要哪些类型的测试数据？
- [ ] 是否需要生产数据脱敏？
- [ ] 是否需要多语言/多区域数据？
- [ ] 是否需要大规模数据集（性能测试）？

#### 3.2 数据准备

- [ ] 是否有现有的测试数据工厂？
- [ ] 是否需要创建新的 Factory/Fixture？
- [ ] 数据清理策略（每个测试后回滚/清理）
- [ ] 数据共享策略（只读数据 vs 可变数据）

**示例**:
```markdown
测试数据需求:
- 用户数据: 使用 UserFactory，支持创建不同角色用户
- 订单数据: 需要关联商品、库存，使用 OrderFactory
- 支付数据: Mock 支付网关，不调用真实接口
- 清理策略: 每个测试后数据库回滚
```

---

### 4. 测试环境需求

#### 4.1 依赖服务

- [ ] 需要哪些外部服务？（数据库、缓存、消息队列）
- [ ] 依赖服务是否可用测试替身（Mock/Stub）？
- [ ] 是否需要 Docker Compose 本地环境？
- [ ] 是否需要共享的测试环境？

#### 4.2 测试替身策略

| 依赖类型 | 推荐策略 | 说明 |
|----------|----------|------|
| 数据库 | Test Container / SQLite 内存 | 真实但隔离 |
| 缓存 | 内存实现 / Test Container | 快速重置 |
| 消息队列 | 内存实现 / Test Container | 验证消息内容 |
| 第三方 API | Mock Server / VCR | 避免外部依赖 |
| 文件存储 | 临时目录 / 内存实现 | 测试后清理 |

---

### 5. 架构可测试性

#### 5.1 测试钩子 (Test Hooks)

- [ ] 是否需要提供测试专用的 API/接口？
- [ ] 是否需要注入测试数据的方式？
- [ ] 是否需要控制时间（冻结时间）的机制？
- [ ] 是否需要模拟外部事件的方式？

**示例**:
```python
# 测试钩子示例
@app.post("/test/seed-data")
def seed_test_data(data: TestData, env: Environment):
    """仅测试环境可用，用于准备测试数据"""
    if env.is_production:
        raise HTTPException(403, "Not allowed in production")
    ...

@app.post("/test/freeze-time")
def freeze_time(timestamp: datetime, env: Environment):
    """冻结时间，用于测试时间相关逻辑"""
    ...
```

#### 5.2 依赖注入

- [ ] 关键依赖是否可注入？（便于 Mock）
- [ ] 配置是否外部化？（便于测试环境配置）
- [ ] 是否有硬编码的依赖？（难以测试的信号）

---

### 6. 自动化测试策略

#### 6.1 测试层级分配

| 需求类型 | 单元测试 | 集成测试 | E2E 测试 | 说明 |
|----------|----------|----------|----------|------|
| 纯计算逻辑 | ✅ 必须 | ❌ 不需要 | ❌ 不需要 | 快速、隔离 |
| 数据库操作 | ⚠️ Mock | ✅ 必须 | ❌ 不需要 | 验证 SQL 正确性 |
| API 接口 | ❌ 不需要 | ✅ 必须 | ⚠️ 关键路径 | 验证契约 |
| 用户流程 | ❌ 不需要 | ❌ 不需要 | ✅ 必须 | 验证完整链路 |
| 第三方集成 | ❌ 不需要 | ✅ 必须 | ❌ 不需要 | 验证适配器 |

#### 6.2 测试优先级

- [ ] P0: 核心业务流程（必须 100% 覆盖）
- [ ] P1: 重要功能路径（目标 90% 覆盖）
- [ ] P2: 边界条件和异常路径（目标 80% 覆盖）
- [ ] P3: 其他场景（根据时间决定）

## 常见问题与解决方案

### 问题 1: 需求过于模糊，无法定义验收标准

**症状**: "系统应该很快"、"用户体验应该良好"

**解决方案**:
1. 要求产品提供量化指标
2. 参考竞品或行业标准
3. 使用用户调研数据支撑

### 问题 2: 依赖外部系统，测试环境难以搭建

**症状**: 需要调用银行 API、政府接口等

**解决方案**:
1. 要求提供 Sandbox 环境
2. 使用 Mock Server（Prism、WireMock）
3. 录制真实响应使用 VCR 模式

### 问题 3: 数据隐私限制，无法使用生产数据测试

**症状**: 涉及 PII、金融数据、医疗数据

**解决方案**:
1. 数据脱敏工具
2. 合成数据生成（Faker + 业务规则）
3. 子集采样（只取 1% 数据）

### 问题 4: 时间敏感逻辑难以测试

**症状**: 限时优惠、定时任务、过期逻辑

**解决方案**:
1. 要求提供时间控制机制
2. 使用 freezegun 等时间冻结工具
3. 抽象时间获取接口（便于 Mock）

## 最佳实践

1. **早期介入**: 在 PRD Draft 阶段就参与，而非等到 Final
2. **协作而非对抗**: 目标是共同提升产品质量，而非挑错
3. **提供解决方案**: 不仅指出问题，还要提供可行的改进建议
4. **记录决策**: 将评审结论和妥协点记录在案
5. **持续跟进**: 开发过程中持续关注可测试性实现
