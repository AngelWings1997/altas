# DEBUG 模式 - 系统化排查专项协议

**触发词**: `DEBUG` / `排查` / `日志分析`

---

## 何时使用

- 用户需要排查 Bug 或异常行为
- 用户需要分析测试失败原因
- 用户需要定位性能瓶颈
- 用户需要理解系统行为（"为什么这样做？"）
- 任何需要**先找根因再修复**的场景

**注意**: 如果是明确的简单修复（如 typo、配置值），且用户明确要求直接修复，可降级为 Coding 流程。但只要涉及"为什么"，必须进入 DEBUG 模式。

**默认规模**: 根据问题复杂度评估（通常 S-M）

---

## 首轮动作

1. **确认问题症状**:
   - 期望行为是什么？
   - 实际观察到什么？
   - 错误信息/日志/堆栈跟踪？
2. **确认复现性**:
   - 能否稳定复现？
   - 复现步骤是什么？
   - 发生频率？（总是/偶尔/难以复现）
3. **确认影响范围**:
   - 影响哪些用户/功能？
   - 是否阻塞主流程？
   - 是否为线上紧急问题？

---

## 排查流程

> **核心协议**: 本模式基于 `references/superpowers/systematic-debugging/SKILL.md` 的四阶段方法论。
> 进入 DEBUG 模式后，**必须先加载**该文档获取完整的 Phase 1-4 详细指导。

### 1) Phase 1: 根因调查（必须完成才能进入修复）

**铁律**: `NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST`

| 步骤 | 动作 | 产出 |
|------|------|------|
| 1.1 | 仔细阅读错误信息 | 错误码、行号、堆栈完整记录 |
| 1.2 | 稳定复现问题 | 复现步骤、频率、环境 |
| 1.3 | 检查最近变更 | Git diff、依赖变更、配置修改 |
| 1.4 | 多组件诊断（如适用） | 各层边界的数据流日志 |

### 2) Phase 2: 假设生成与验证

- 基于 Phase 1 证据，提出 2-3 个根因假设
- 对每个假设设计验证实验
- 收集证据排除或确认假设
- **禁止在根因未明时进入修复**

### 3) Phase 3: 修复方案设计

- 确认根因后，设计最小化修复方案
- 评估修复的影响面和风险
- 若影响面大，考虑是否需要升级为 REFACTOR 或 MIGRATE

### 4) Phase 4: 修复与验证

- 执行修复
- 运行回归测试验证
- 确认问题已解决且无副作用
- 输出排查报告

---

## 输出产物

**排查报告模板**:

```markdown
## 排查报告

**问题日期**: <YYYY-MM-DD>
**问题描述**: <一句话描述>

### 症状
- **期望行为**: <应该发生什么>
- **实际行为**: <实际发生了什么>
- **错误信息**: <完整错误/日志>

### 复现性
- **能否稳定复现**: 是/否/偶尔
- **复现步骤**:
  1. ...
  2. ...
- **发生频率**: 总是 / X% 概率 / 难以复现

### 根因分析
- **根因**: <一句话总结根本原因>
- **证据链**:
  1. <证据 1>
  2. <证据 2>
  → 结论: <根因假设>
- **排除的假设**:
  - ❌ <假设 A>: <排除理由>
  - ❌ <假设 B>: <排除理由>

### 修复方案
- **修复内容**: <做了什么改动>
- **影响范围**: <影响了哪些模块/功能>
- **回归测试**: <运行了什么测试>

### 经验教训 (可选)
- <如何避免类似问题再次发生>
```

---

## 门禁逻辑

| 场景 | 处理方式 |
|------|----------|
| 根因未明就想改代码？ | **禁止**。回到 Phase 1 继续调查 |
| 问题无法复现？ | 先增加日志/监控，收集更多数据，不要猜测 |
| 发现多个可能根因？ | 逐一验证，用证据排除，不要凭直觉选一个 |
| 修复后未验证？ | 必须运行测试或手动验证确认问题已解决 |
| 排查时间过长？ | 输出检查点，说明当前进度和下一步计划 |

## 与其他模式协作

- **DEBUG → TEST**: 排查后发现是测试缺陷（断言错误/Mock 问题），进入 TEST 模式修正测试
- **DEBUG → REVIEW**: 根因复盘，进入 REVIEW 模式审查相关代码质量
- **DEBUG → PERF**: 排查发现是性能问题（慢查询/内存泄漏），进入 PERF 模式优化
- **DEBUG → REFACTOR**: 根因是代码坏味道（过度耦合/复杂度高），进入 REFACTOR 模式重构

## 特殊场景处理

### 场景 1: 线上紧急故障（P0）

- **快速通道**: 跳过详细文档，聚焦根因定位
- **时间约束**: 设定最大排查时间（如 30 分钟），超时后升级或回滚
- **并行行动**: 可同时进行临时缓解（重启/降级）+ 根因排查
- **事后复盘**: 即使紧急修复完成，也建议后续输出简化版排查报告

### 场景 2: 难以复现的间歇性问题

- 增加诊断日志到可疑代码路径
- 使用监控/APM 工具持续观察
- 尝试构造压力测试复现条件
- 若仍无法复现，标记为"待观察"并定期回顾

### 场景 3: 第三方依赖问题

- 先隔离：确认问题是自身代码还是外部依赖
- 查阅依赖的 changelog/issue tracker
- 尝试升级/降级依赖版本验证
- 若确认为依赖 Bug，提交 issue 并寻找 workaround

### 场景 4: 数据库/存储相关问题

- 先备份当前状态（防止排查过程中数据丢失）
- 使用只读查询验证假设
- 避免在生产环境执行破坏性操作
- 数据修复必须有回滚方案
