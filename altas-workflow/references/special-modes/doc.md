# DOC 模式 - 文档撰写专项协议

**触发词**: `DOC` / `写文档` / `文档`

---

## 何时使用

- 用户需要为代码/功能/项目撰写技术文档
- 用户需要更新现有文档（API 文档/README/架构文档）
- 用户需要生成设计文档或决策记录
- 用户需要将代码逻辑转化为人类可读的说明

**注意**: 如果只是简单的注释补充或 README 更新（< 50 行），可降级为 Coding 流程的 XS 规模处理。但只要涉及"结构化文档产出"，必须进入 DOC 模式。

**默认规模**: 根据文档复杂度评估（通常 S-L）

---

## 首轮动作

1. **确认文档目标**:
   - 文档类型？（API 文档/架构文档/用户指南/CHANGELOG/...）
   - 目标读者？（开发者/运维/产品经理/最终用户）
   - 输出格式？（Markdown/LaTeX/HTML/...）
2. **确认文档范围**:
   - 涉及哪些模块/功能？
   - 需要覆盖到什么深度？
   - 是否有现有文档需要更新？
3. **确认约束**:
   - 是否需要遵循特定文档风格？
   - 是否有模板要求？
   - 语言要求？（中文/英文/双语）

## 文档流程

> **核心协议**: 本模式基于 `protocols/RIPER-DOC.md` 的四阶段方法论。
> 进入 DOC 模式后，**必须遵循** ABSORB → OUTLINE → AUTHOR → FACT-CHECK 的顺序执行。

### 1) MODE: ABSORB — 上下文提取

**目标**: 理解代码逻辑，收集技术细节，**暂不写文案**

| 动作 | 说明 | 产出 |
|------|------|------|
| 读取源码 | 理解参数、返回值、逻辑流 | 关键技术细节清单 |
| 识别边界 | 找出异常路径、特殊条件 | 边界情况列表 |
| 收集示例 | 提取典型使用场景 | 使用案例 |

**约束**: 
- **禁止猜测代码行为**，必须验证
- 只输出技术要点，不输出 prose
- 记录不确定项，后续澄清

### 2) MODE: OUTLINE — 结构规划

**目标**: 设计文档目录结构

| 动作 | 说明 | 产出 |
|------|------|------|
| 设计 TOC | H1/H2/H3 层级结构 | 目录大纲 |
| 风格适配 | 匹配项目现有文档风格 | 风格指南确认 |
| 内容分配 | 各章节预计内容量 | 写作计划 |

**约束**:
- 必须符合项目的文档规范
- 结构应便于后续维护和更新
- 考虑读者的阅读路径

### 3) MODE: AUTHOR — 内容生成

**目标**: 基于技术要点和结构，撰写实际文档内容

| 动作 | 说明 | 产出 |
|------|------|------|
| 撰写正文 | 清晰、简洁、专业 | 完整草稿 |
| 编写示例 | 可运行的代码示例 | 示例集 |
| 补充图表 | 架构图/流程图/时序图 | 可视化素材 |

**风格要求**:
- 使用清晰的语言，避免歧义
- 技术术语首次出现时给出解释
- 保持一致的术语和格式
- 适当使用表格、列表提高可读性

### 4) MODE: FACT-CHECK — 准确性验证

**目标**: 交叉验证文档与实际代码的一致性

**检查清单**:
- [ ] 所有参数名是否与代码一致？
- [ ] 默认值是否准确？
- [ ] 代码示例是否可运行？
- [ ] 返回值描述是否正确？
- [ ] 异常场景是否覆盖？

**输出**: `✅ DOCS VERIFIED` 或 `⚠️ CORRECTION MADE`（附修改说明）

## 输出产物

根据文档类型选择合适的模板：

### API 文档模板
```markdown
# <API 名称> 文档

## 概述
<一句话描述 API 的功能>

## 端点
| 方法 | 路径 | 描述 |
|------|------|------|

## 请求/响应
### 请求参数
| 参数 | 类型 | 必填 | 描述 |
|------|------|------|------|

### 响应结构
<JSON 示例>

## 错误码
| 状态码 | 含义 | 解决方案 |
|--------|------|----------|

## 示例
<curl / Python / JavaScript 示例>
```

### 架构文档模板
```markdown
# <系统/模块> 架构文档

## 背景
<为什么这样设计>

## 整体架构
<架构图>

## 核心组件
### <组件 1>
<职责/接口/依赖>

### <组件 2>
...

## 数据流
<请求处理流程图>

## 技术选型
<为什么选择这些技术>

## 部署拓扑
<部署图>
```

## 门禁逻辑

| 场景 | 处理方式 |
|------|----------|
| 未读代码就开始写文档？ | **禁止**。回到 ABSORB 模式先理解代码 |
| 文档与代码不一致？ | 回到 FACT-CHECK 修正，或重新 ABSORB |
| 发现代码本身有问题？ | 暂停文档工作，切换到 DEBUG 或 REVIEW 模式 |
| 文档范围蔓延？ | 输出检查点，确认是否需要升级规模或拆分任务 |
| 缺少示例？ | 必须补充至少一个可运行示例 |

---

## 与其他模式协作

- **DOC → REVIEW**: 文档完成后，进入 REVIEW 模式审查文档质量（准确性/完整性/可读性）
- **DOC → ARCHIVE**: 文档作为知识沉淀的一部分，进入 ARCHIVE 模式归档
- **DOC → DEBUG**: 撰写过程中发现代码行为异常，暂停文档工作进入 DEBUG 排查

## 特殊场景处理

### 场景 1: 为遗留系统补文档

- 优先从代码中提取信息，而非依赖口头描述
- 对于不清晰的逻辑，标注 `[待确认]` 并联系原作者
- 分阶段交付：先出骨架，再逐步完善

### 场景 2: 多语言文档（双语）

- 先完成主语言版本（通常是英文）
- 再翻译为次要语言
- 使用翻译工具辅助，但人工审核专业术语
- 维护两个版本的同步性

### 场景 3: 文档驱动开发 (DDD)

- 在编码前先写文档（接口文档/设计文档）
- 文档作为 Spec 的一部分
- 编码完成后更新文档以反映实现细节

### 场景 4: 自动化文档生成

- 从代码注释/类型签名自动提取 API 文档框架
- 从 OpenAPI/Swagger 生成交互式文档
- 从测试用例生成使用示例
- 自动化部分 + 手工润色的混合模式
