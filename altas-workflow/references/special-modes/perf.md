# PERF 模式 - 性能优化专项协议

**触发词**: `PERF` / `性能优化`

---

## 何时使用

- 用户需要优化代码性能（响应时间/吞吐量/资源消耗）
- 用户需要排查性能瓶颈
- 用户需要性能基准测试
- 用户需要容量规划建议

**默认规模**: L（性能优化需要基准测试→定位→优化→验证的完整闭环）

---

## 首轮动作

1. **确认性能目标**:
   - 优化哪个指标？（响应时间/吞吐量/CPU 使用率/内存占用/...）
   - 目标值是多少？（如"响应时间从 500ms 降到 100ms"）
2. **确认优化范围**:
   - 单个函数/接口？
   - 整个系统？
3. **确认约束**:
   - 是否允许改动架构？
   - 是否允许增加资源（如缓存/CDN）？
   - 是否有时间窗口限制？

---

## 性能优化流程

### 1) 基准测试（必须）

**在优化前，必须建立性能基线**:

- 定义性能指标（如 P95 响应时间/QPS/内存峰值）
- 定义测试场景（如"100 并发用户持续 60 秒"）
- 运行基准测试，记录基线数据
- 输出性能基准报告

**基准报告模板**:

```markdown
## 性能基准报告

**测试日期**: <YYYY-MM-DD>
**测试场景**: <场景描述>
**测试工具**: <工具名，如 wrk/k6/JMeter>

### 基线数据
| 指标 | 值 | 单位 |
|------|-----|------|
| P50 响应时间 | X | ms |
| P95 响应时间 | X | ms |
| P99 响应时间 | X | ms |
| 吞吐量 | X | req/s |
| CPU 使用率 | X | % |
| 内存占用 | X | MB |
| 错误率 | X | % |

### 测试命令
<运行基准测试的完整命令>
```

### 2) Profile 定位瓶颈

**使用 Profiling 工具定位性能瓶颈**:

| 语言 | 推荐工具 |
|------|----------|
| JavaScript/Node.js | Chrome DevTools, clinic.js, 0x |
| Python | cProfile, py-spy, line_profiler |
| Go | pprof, trace |
| Java | VisualVM, JProfiler, async-profiler |
| Rust | perf, flamegraph |

**输出瓶颈分析报告**:

```markdown
## 瓶颈分析报告

### 热点函数
| 函数名 | 调用次数 | 耗时占比 | 文件位置 |
|--------|----------|----------|----------|
| <函数 1> | X | Y% | file.ts:L123 |
| <函数 2> | X | Y% | file.ts:L456 |

### 瓶颈类型
- [ ] CPU 密集型（计算密集）
- [ ] I/O 密集型（数据库/网络请求）
- [ ] 内存密集型（频繁分配/GC 压力）
- [ ] 锁竞争（并发瓶颈）

### 根因分析
<详细描述瓶颈的根本原因>
```

### 3) 优化方案对比

针对每个瓶颈，给出 2-3 种优化方案：

| 方案 | Pros | Cons | 预计收益 | 实现成本 |
|------|------|------|----------|----------|
| 方案 1 | ... | ... | -X% 耗时 | 低/中/高 |
| 方案 2 | ... | ... | -Y% 耗时 | 低/中/高 |

**常见优化策略**:

| 瓶颈类型 | 优化策略 |
|----------|----------|
| **CPU 密集型** | 算法优化、缓存结果、并行计算、WebAssembly |
| **I/O 密集型** | 批量请求、连接池、缓存、异步化 |
| **内存密集型** | 对象复用、流式处理、数据结构优化 |
| **锁竞争** | 无锁数据结构、细粒度锁、读写锁 |

### 4) 制定优化计划

**计划必须包含**:

- **Checklist**: 原子优化步骤（每步独立可验证）
- **验证策略**: 每步完成后如何测量性能提升
- **回滚方案**: 若优化失败或引入 Bug，如何快速回退

**门禁**: 计划必须获得用户 `[Approved]` 后方可执行

### 5) 小步执行（优化→验证循环）

对每个 Checklist 项：

```
1. 执行优化
2. 运行基准测试（与基线对比）
3. 验证功能正确性（运行现有测试）
4. 若性能提升且无 Bug → 输出检查点，进入下一步
5. 若性能未提升或引入 Bug → 回滚或调整方案
```

**铁律**:
- 每次只应用一个优化（防止多个优化互相干扰）
- 每个优化必须有可量化的性能提升
- 不允许以牺牲正确性为代价换取性能

### 6) 优化后验证

- 运行完整基准测试（与基线对比）
- 运行完整测试套件（确保无功能退化）
- 输出性能优化报告

**优化报告模板**:

```markdown
## 性能优化报告

**优化日期**: <YYYY-MM-DD>
**优化目标**: <如"P95 响应时间从 500ms 降到 100ms">

### 优化前后对比
| 指标 | 优化前 | 优化后 | 变化 |
|------|--------|--------|------|
| P50 响应时间 | X ms | Y ms | -Z% |
| P95 响应时间 | X ms | Y ms | -Z% |
| P99 响应时间 | X ms | Y ms | -Z% |
| 吞吐量 | X req/s | Y req/s | +Z% |
| CPU 使用率 | X% | Y% | -Z% |
| 内存占用 | X MB | Y MB | -Z% |

### 优化措施清单
1. [措施 1 描述] - 收益：-X% 耗时
2. [措施 2 描述] - 收益：-Y% 耗时
...

### 未达标项 (若有)
- <指标名>: 当前值 X，目标值 Y
- 原因分析: ...
- 后续建议: ...

### 性能预算建议 (可选)
- 建议设置性能预算，防止性能退化
- 建议将基准测试纳入 CI/CD
```

---

## 门禁逻辑

| 场景 | 处理方式 |
|------|----------|
| 优化后性能未提升 | 回滚优化，重新分析瓶颈 |
| 优化后引入 Bug | 回滚优化，修复 Bug 后重新尝试 |
| 优化后其他指标退化 | 权衡利弊，用户决定是否接受 |
| 优化未达标 | 若距离目标较远，重新 Profile 寻找新瓶颈 |

---

## 与其他模式协作

- **PERF → DEBUG**: 优化后出现异常行为，进入 DEBUG 模式排查
- **PERF → REVIEW**: 优化完成后，进入 REVIEW 模式审查优化质量
- **PERF → ARCHIVE**: 优化经验沉淀，进入 ARCHIVE 模式生成性能优化知识库

---

## 特殊场景处理

### 场景 1: 性能问题偶发（难以复现）

- 建议用户先建立稳定的复现环境
- 若无法稳定复现，建议使用 APM 工具（如 New Relic/DataDog）持续监控

### 场景 2: 性能瓶颈在外部依赖（数据库/第三方 API）

- 建议优化方向：
  - 缓存外部依赖的结果
  - 批量请求减少网络往返
  - 异步化不阻塞主流程
  - 与外部依赖提供方协商优化

### 场景 3: 性能优化 vs 代码可读性冲突

- 若优化导致代码可读性严重下降，输出提示：
  - "该优化会降低代码可读性，建议添加详细注释说明优化原因"
  - 或提议替代方案（如提取为独立函数并命名清晰）
