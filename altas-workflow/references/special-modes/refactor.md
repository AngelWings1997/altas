# REFACTOR 模式 - 重构专项协议

**触发词**: `REFACTOR` / `重构`

---

## 何时使用

- 用户需要改进代码结构而不改变外部行为
- 用户需要消除技术债务/代码坏味道
- 用户需要为新功能铺路（先重构再扩展）
- 用户需要优化代码可读性/可维护性

**默认规模**: M/L（重构涉及理解现有代码 + 小步验证）

---

## 首轮动作

1. **确认重构目标**: 
   - 消除哪个/哪些代码坏味道？
   - 提升哪方面质量？（可读性/可维护性/可扩展性/性能）
2. **确认重构范围**: 
   - 单个函数/类/模块？
   - 跨模块重构？
3. **确认约束**:
   - 是否允许改动公共 API？
   - 是否需要向后兼容？
   - 是否有时间窗口限制？

---

## 重构流程

### 1) CodeMap 先行（必须）

- 生成待重构代码的 CodeMap
- 标识：
  - 入口点（调用方）
  - 依赖项（被依赖的代码）
  - 测试覆盖情况
  - 风险点（复杂度高/改动影响大的区域）

### 2) 坏味道识别

参考常见代码坏味道清单：

| 坏味道 | 描述 | 重构策略 |
|--------|------|----------|
| **重复代码** | 相同/相似代码出现在多处 | 提取函数/提取公共模块 |
| **过长函数** | 函数超过 50 行 | 提取函数/分解为多个小函数 |
| **过大类** | 类超过 500 行或职责过多 | 拆分类/提取职责 |
| **过长参数列表** | 函数参数超过 5 个 | 引入参数对象/使用配置对象 |
| **过度耦合** | 模块间依赖复杂 | 引入接口/依赖注入 |
| **命名不清晰** | 变量/函数名不能表达意图 | 重命名 |
| **条件复杂** | 嵌套 if/switch 超过 3 层 | 提取条件函数/使用策略模式 |
| **数据簇** | 多个数据总是一起出现 | 封装为数据结构/类 |

### 3) 制定重构计划

**计划必须包含**:

- **Checklist**: 原子重构步骤（每步改动 < 50 行）
- **回归测试策略**: 每步完成后如何验证行为不变
- **回滚方案**: 若某步失败，如何快速回退

**门禁**: 计划必须获得用户 `[Approved]` 后方可执行

### 4) 小步执行（TDD 循环）

对每个 Checklist 项：

```
1. 确认当前测试全部通过（建立基线）
2. 执行小步重构
3. 运行回归测试
4. 若测试失败 → 修复或回滚
5. 若测试通过 → 输出检查点，请求进入下一步
```

**铁律**: 
- 每步重构后必须验证行为不变
- 不允许在重构中混入新功能开发（除非用户明确要求）
- 若发现必须改动行为 → 暂停，回到 Plan 重新对齐

### 5) 重构后审查

- CodeMap 对比（重构前 vs 重构后）
- 指标对比（如函数行数/圈复杂度/依赖数）
- 测试覆盖率对比

## 输出产物

| 产物 | 路径 |
|------|------|
| CodeMap(重构前) | `mydocs/codemap/YYYY-MM-DD_hh-mm_<模块>_重构前.md` |
| CodeMap(重构后) | `mydocs/codemap/YYYY-MM-DD_hh-mm_<模块>_重构后.md` |
| 重构计划 | Spec 中的 Plan 部分 |
| 重构记录 | `mydocs/refactor/YYYY-MM-DD_hh-mm_<模块>.md` (可选) |

**重构记录模板**:

```markdown
# 重构记录

**目标模块**: <模块名>
**重构日期**: <YYYY-MM-DD>
**重构目标**: <消除坏味道/提升可维护性/...>

## 重构前后对比

### CodeMap 对比
- 重构前：<链接>
- 重构后：<链接>

### 指标对比
| 指标 | 重构前 | 重构后 | 变化 |
|------|--------|--------|------|
| 函数平均行数 | X | Y | -Z% |
| 圈复杂度 | X | Y | -Z% |
| 测试覆盖率 | X | Y | +Z% |

## 重构步骤清单
1. [步骤 1 描述] - [PASS/FAIL]
2. [步骤 2 描述] - [PASS/FAIL]
...

## 经验总结 (可选)
- 哪些重构策略有效
- 哪些坑需要避免
```

---

## 门禁逻辑

| 场景 | 处理方式 |
|------|----------|
| 重构中发现 Bug | 暂停重构 → 记录 Bug → 用户确认是否先修复 Bug 还是继续重构 |
| 重构后测试失败 | 必须修复或回滚，不得带着失败测试进入下一步 |
| 重构偏离计划 | 若偏差 > 2 次 → 暂停，重新 Plan |
| 重构引入破坏性变更 | 立即暂停 → 评估影响面 → 用户确认是否继续 |

---

## 与其他模式协作

- **REFACTOR → TEST**: 重构后发现测试覆盖不足，进入 TEST 模式补测试
- **REFACTOR → DEBUG**: 重构后出现意外行为，进入 DEBUG 模式排查
- **REFACTOR → REVIEW**: 重构完成后，进入 REVIEW 模式审查重构质量

---

## 特殊场景处理

### 场景 1: 大重构（跨多模块）

- 必须拆分为多个独立子任务
- 每个子任务有独立的 CodeMap + Plan + 回归测试
- 建议按依赖顺序执行（先重构底层依赖，再重构上层）

### 场景 2: 无测试覆盖的老代码

- 重构前必须先补关键路径测试（至少覆盖核心行为）
- 若无法补测试（如依赖复杂/环境限制），输出风险提示，建议用户先补测试再重构

### 场景 3: 重构 vs 重写边界

- 若重构成本 > 重写成本（如代码质量极差/技术债务过重），建议用户选择重写而非重构
- 判断标准：重构步骤 > 20 步 或 预计改动 > 1000 行 → 提议重写
