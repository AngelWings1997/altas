# REVIEW 模式 - 代码审查专项协议

**触发词**: `REVIEW` / `代码审查` / `审查 PR`

---

## 何时使用

- 用户需要审查 Pull Request / Merge Request
- 用户需要审查代码变更（diff）
- 用户需要第三方视角的代码质量评估
- 用户需要审查 Spec 与代码的一致性

**注意**: 如果审查是完整开发流程的最后阶段（Execute 之后），使用标准工作流的 Review 阶段即可，不需要单独触发 REVIEW 模式。

---

## 首轮动作

1. **确认审查范围**: 文件列表 / PR 链接 / 代码 diff
2. **确认审查目标**: 找 Bug / 代码质量 / 架构合理性 / 安全漏洞 / 测试覆盖
3. **确认审查深度**: 
   - **Lite**: 快速扫描，只输出高风险问题
   - **Standard**: 完整三轴评审
   - **Deep**: 逐行审查 + 重构建议

---

## 审查流程

### 1) 准备阶段

- 读取待审查的代码 / PR diff
- 若存在关联 Spec，读取 Spec 作为审查依据
- 若存在关联 Plan/Checklist，核对完成情况

### 2) 三轴评审（Standard/Deep 必须）

| 轴 | 检查项 | 输出格式 |
|----|--------|----------|
| **Spec 质量与需求达成** | Goal/In-Scope/Acceptance 是否完整；需求是否达成 | PASS/FAIL/PARTIAL + 依据 |
| **Spec-代码一致性** | 文件、签名、Checklist、行为是否与 Plan 一致 | PASS/FAIL/PARTIAL + 差异列表 |
| **代码内在质量** | 正确性、鲁棒性、可维护性、测试、关键风险 | PASS/FAIL/PARTIAL + 问题清单 |

### 3) 问题分级

| 级别 | 定义 | 示例 |
|------|------|------|
| **P0 - 阻塞** | 必须修复，否则不能合并 | 安全漏洞、核心逻辑错误、数据不一致 |
| **P1 - 高优** | 强烈建议修复 | 未处理的异常路径、关键逻辑无测试、性能瓶颈 |
| **P2 - 建议** | 可择机优化 | 代码风格不统一、注释缺失、可读性差 |
| **P3 - 可选** | 锦上添花 | 命名不够精确、小的重构机会 |

### 4) 输出审查报告

**标准格式**:

```markdown
## 审查报告

**审查范围**: <文件列表/PR 链接>
**审查深度**: <Lite/Standard/Deep>
**审查目标**: <找 Bug/代码质量/架构合理性/...>

### 总体评价
- 轴 1 (Spec 质量): PASS/FAIL/PARTIAL
- 轴 2 (一致性): PASS/FAIL/PARTIAL
- 轴 3 (代码质量): PASS/FAIL/PARTIAL
- **综合结论**: [建议合并 / 修复后合并 / 需要重大修改 / 拒绝]

### 问题清单

#### P0 - 阻塞 (X 个)
1. [文件：行号] 问题描述 + 修复建议
2. ...

#### P1 - 高优 (X 个)
1. [文件：行号] 问题描述 + 修复建议
2. ...

#### P2 - 建议 (X 个)
...

#### P3 - 可选 (X 个)
...

### 亮点 (可选)
- 值得肯定的设计/实现
```

---

## 门禁逻辑

| 审查结果 | 后续动作 |
|----------|----------|
| 轴 1 或轴 2 = FAIL | 回到 Research/Plan 修正，不得合并 |
| 轴 3 有 P0 问题 | 必须修复后重新审查 |
| 轴 3 有 P1 问题 | 建议修复，用户可决定是否阻塞合并 |
| 全部 PASS 或仅有 P2/P3 | 建议合并 |

---

## 与其他模式协作

- **REVIEW → REFACTOR**: 审查发现重构机会，用户确认后进入 REFACTOR 模式
- **REVIEW → DEBUG**: 审查发现潜在 Bug，用户确认后进入 DEBUG 模式排查
- **REVIEW → TEST**: 审查发现测试覆盖不足，用户确认后进入 TEST 模式补测试

---

## 特殊场景处理

### 场景 1: 审查外部代码（无 Spec）

- 重点审查：代码内在质量轴（轴 3）
- 输出：问题清单 + 重构建议 + 风险提示
- 不强制要求 Spec 一致性审查

### 场景 2: 审查自己的代码（自我审查）

- 建议用户明确要求审查目标（如"重点看有没有边界情况遗漏"）
- 输出风格：建议式而非审判式

### 场景 3: 审查超大 PR（>500 行）

- 建议拆分为多个逻辑单元分别审查
- 若无法拆分，输出声明："由于 PR 规模过大，以下审查可能不完整，建议拆分后重新审查"
