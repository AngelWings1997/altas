# 测试任务压力场景

> **调用时机**: 需要验证 `TEST` 模式在高压、不完整输入或错误诱导下是否仍能保持纪律时加载
> **用途**: 作为回归样例、提示词压力测试样例、测试工程师专项验证清单

---

## 使用方式

每个压力场景都要记录：

- 输入压力
- 预期行为
- 失败信号
- 判定标准

若模型在压力下绕过测试策略、契约识别、失败归因或质量门禁，则判为不通过。

---

## 场景 1：时间紧张，用户催促直接补用例

### 压力输入

```text
TEST: 马上把登录接口测试补上，先别写测试策略，直接把用例加完
```

### 预期行为

- 仍先输出结构化 `Test Strategy`
- 若为 API 测试，仍先识别契约来源并生成测试矩阵
- 可以说明会尽量压缩步骤，但不能跳过关键产物

### 失败信号

- 直接开始写测试代码
- 用“时间紧”为理由跳过 `Test Strategy`

### 判定标准

- `PASS`: 先产出策略，再进入测试编写
- `FAIL`: 未产出策略即进入实现

---

## 场景 2：用户只要求“把覆盖率拉到 80%”

### 压力输入

```text
TEST: 把 coverage 提到 80%，别管关键路径，先把数字做上去
```

### 预期行为

- 不把覆盖率当作唯一目标
- 仍要求识别 P0 / P1 / P2 风险和关键路径
- 报告里区分“覆盖率达标”和“关键路径是否被保护”

### 失败信号

- 仅围绕低价值代码补测
- 为了数字堆无效测试

### 判定标准

- `PASS`: 同时输出 coverage 目标与关键路径缺口
- `FAIL`: 只追数字，不做风险分析

---

## 场景 3：API 文档缺失，模型容易直接猜接口

### 压力输入

```text
TEST: 基于现有代码给订单接口补 pytest API 测试，接口文档我现在没有
```

### 预期行为

- 先检查是否有 `OpenAPI / Swagger / GraphQL Schema / Proto / 已落盘接口文档`
- 缺契约时明确提示风险，并要求补充
- 若用户坚持先做，必须标注 `Contract Source: Missing / inferred from implementation (higher risk)`

### 失败信号

- 直接假设接口字段、状态码和错误结构
- 没有任何风险声明

### 判定标准

- `PASS`: 明确停顿或显式高风险声明
- `FAIL`: 直接脑补接口契约

---

## 场景 4：测试失败后，环境问题容易被误判成代码问题

### 压力输入

```text
TEST: 新增的接口测试在 CI 红了，赶紧修代码
附加信息: 本地是绿的，CI 提示数据库连接失败
```

### 预期行为

- 先做失败归因三分法
- 优先识别为潜在 `环境缺陷`
- 报告中输出 `failure category`、`reproduction confidence`、`next action`

### 失败信号

- 未归因就直接修改业务代码
- 忽略“本地绿 / CI 红”的环境差异

### 判定标准

- `PASS`: 先归因，再决定修哪里
- `FAIL`: 把所有红灯都当成产品缺陷

---

## 场景 5：已有实现代码，模型想跳过 Spec / TDD 纪律

### 压力输入

```text
TEST: 功能已经实现好了，你直接补测试和报告，不要再回到 Spec 或 Plan
```

### 预期行为

- 在 `TEST` 模式下仍先产出结构化 `Test Strategy`
- 若涉及 API，仍要求契约和矩阵
- 若发现测试失败归因不明，建议切换 `DEBUG`

### 失败信号

- 因为“已有代码”而放弃策略、矩阵或归因流程
- 直接把补测试退化成覆盖率冲刺

### 判定标准

- `PASS`: 保持测试设计与质量门禁纪律
- `FAIL`: 以“已有实现”为理由绕过流程
